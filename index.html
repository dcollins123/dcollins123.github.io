01 — Professional Self-Assessment

What I got out of three and a half years.

I started the Computer Science program at SNHU in August 2022 and I'm wrapping up in April 2026. Somewhere in the middle of all that, the stuff I was studying actually started sticking. This ePortfolio is basically me showing that.

The three artifacts here aren't my biggest projects from the program, but they're the ones that fit the three capstone categories the best. Each one started as a finished class project and got a pretty big rebuild for this. Together they cover most of what a software engineer actually does: shipping something people can use, picking the right algorithm for the job, and building a backend that doesn't leak.

Collaborating with others

A lot of the program was group stuff. Team projects, code reviews, pull requests, arguing about variable names (or at least attempting to). CS 250 (Software Development Lifecycle) beat Agile and Scrum into me pretty hard, and that stuff showed up in basically every group project after. The code review video below is probably the clearest example here — me walking through three codebases and explaining things out loud to someone who wasn't in my head when I wrote any of it.

Communicating professionally

Writing is not optional if you want your code to go anywhere. Every artifact here has a README explaining what I changed and why. The code review is me talking about the same stuff out loud. And this self-assessment is kind of tying the room together. Same goal each time — making the work make sense to someone who wasn't there.

Algorithmic thinking and trade-offs

CS 370 is the cleanest example of this one. The original project used deep Q-learning to solve an 8x8 maze, which works but is overkill for a deterministic shortest-path problem. Like using a flamethrower to light a candle. I added A* and BFS next to the Q-learning agent and benchmarked all three. A* and BFS both found the optimal 24-step path in under a millisecond. Q-learning took over an hour of training to get to about 40% win rate. The point isn't which one "wins" — it's showing that I get when reinforcement learning is actually worth it (stochastic stuff, unknown dynamics, huge state spaces) versus when it's the wrong tool. CS 260 (Data Structures and Algorithms) was what made this comparison possible in the first place.

Well-founded, innovative techniques and tools

Across the three enhancements I used Three.js, WebGL, FastAPI, Pydantic, MongoDB aggregation pipelines, JWT, bcrypt, Leaflet, Plotly, and probably some other stuff I'm forgetting. Each choice was on purpose though — Three.js because it's basically the standard for 3D on the web, FastAPI because it hands you auto-generated docs and type-checked validation for free, MongoDB aggregation because it lets the database do the heavy lifting instead of dragging everything into Python. The portfolio itself is on GitHub Pages with each artifact in its own repo, so everything is public and version-controlled.

Security mindset

This one mostly involves the CS 340 enhancement. The original dashboard had zero authentication. Like, none. Anyone who could reach the notebook could reach the database. The enhanced version has JWT tokens, bcrypt-hashed passwords, role-based access with three roles (admin/write/read), Pydantic validation on every endpoint, and NoSQL injection sanitization that strips Mongo operators out of incoming queries. CS 405 (Secure Coding) and IT 380 (Cybersecurity) kind of drilled this into me — security isn't something you duct tape on at the end, it's supposed to be the default.

Where I'm headed

Full-stack software engineering is the goal. The three artifacts below were picked specifically to show that — a front-end rendering thing (CS 330), an algorithms thing (CS 370), and a secure backend with a real database (CS 340). If someone asks whether I can work across the stack, I'd rather point at this page than try to explain it.

Took a Jupyter/Dash dashboard and rebuilt it as a FastAPI backend with a standalone HTML/JS frontend.
